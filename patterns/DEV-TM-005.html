DEV-TM-005: Fragile Base Class による骨格の崩壊

C: 侵食・崩壊（Erosion / Decay） 逸脱タイプ: 構造 根拠: 4件 合議: agree
逸脱パターン名Fragile Base Class による骨格の崩壊
対象デザインパターンTemplate Method
逸脱タイプ構造
A/B/C 判定C: 侵食・崩壊（Erosion / Decay）
一般形式AbstractClass の TemplateMethod() または PrimitiveOperation() の実装を変更したとき、その変更が ConcreteClass の動作を意図せず壊す（Fragile Base Class 問題）
症状と結果ベースクラスの「安全に見える変更」（メソッドの実装変更・呼び出し順序の変更等）が、サブクラスの override したメソッドと組み合わさって誤動作を引き起こす。例：storeDocument() の変更により、EncryptedStorage のサブクラスが暗号化処理をバイパスする。
解決策Template Method の PrimitiveOperation() 呼び出し順序や事前条件を変更するときは、全 ConcreteClass への影響を検証する。open/closed principle を守り、変更ではなく拡張でアルゴリズムを変化させる設計にする。
防止策TemplateMethod() の内部実装を変更する際はリグレッションテストを必ず実行する。継承で使う予定のあるクラスはインターフェースを安定させ、変更不可部分を final にする。Joshua Bloch「継承のために設計・文書化するか、さもなくば禁止せよ」の原則に従う。
逸脱例Bag クラスの addAll() が add() を n 回呼ぶ実装から一括追加に変更されたとき、CountingBag（add() をカウント付き override）が追加件数を二重計上する。
