DEV-TM-011: Template Method適用境界の誤認（コンポジションが適切な場面での使用）

A: 意図保持の変形（Adaptive Variant） 逸脱タイプ: 目的 根拠: 3件 合議: agree
逸脱パターン名Template Method適用境界の誤認（コンポジションが適切な場面での使用）
対象デザインパターンTemplate Method
逸脱タイプ目的
A/B/C 判定A: 意図保持の変形（Adaptive Variant）
一般形式アルゴリズムの変形がコンポジション（Strategy, Decorator, ラムダ等）で十分解決できる場面で、Template Method（継承ベース）を選択する
症状と結果Template Method の継承によるサブクラスが増加し、組み合わせの爆発が起きる。実行時にアルゴリズムを切り替えたい場合に対応できない（継承は静的）。Effective Java 等でコンポジション優先が推奨される場面での使用。
解決策Strategy パターン、ラムダ式、Builder、Decorator 等に移行する。特に Java 8 以降では、abstract メソッドを関数型インターフェースへの引数に置き換えられる。
防止策Template Method 適用時に「実行時に振る舞いを切り替える必要があるか」「is-a 関係か has-a 関係か」を判断する。has-a 関係なら必ずコンポジションを選択する。
逸脱例ソートアルゴリズムの変更を AbstractSorter サブクラスで対応する設計を採用したが、後から「実行時にアルゴリズムを動的に切り替えたい」という要件が出てきて対応できなくなった。
関連する逸脱パターン
DEV-TM-006, DEV-TM-007
備考A判定（意図保持の変形）：GoFの意図から外れるが、継承よりコンポジションを選ぶ判断はアーキテクチャ上の合理的選択で有害ではない。ただし意図的な代替選択であればパターンの「誤用」ではなく「意図的な非採用」。
